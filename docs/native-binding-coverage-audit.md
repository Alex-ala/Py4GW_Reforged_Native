# Native binding-coverage audit (reforged GW modules)

Date: 2026-07-04. Generated by a per-module fan-out audit (native-declared API vs
Python-bound surface vs legacy-library need). See memory `native-binding-coverage-gap`.

## APPROACH CORRECTION (owner, 2026-07-04) - read before using the plan below

Reading game DATA via the Python ctypes contexts is a DELIBERATE design decision
(ctypes conversion is FASTER than pybind marshaling). So this audit's recurring advice
to "bind the Context* data accessors as py::class_" is WRONG for this codebase and must
be IGNORED. The real gap is OPERATIONS the legacy bindings exposed that the reforged
bindings dropped - above all async string DECODING (ctypes cannot decode GW encoded
strings; the lib helper encoded_wstr_to_str is only an escaper). Method:

  Per module, diff legacy `Py4GW/src/py_<module>.cpp` against reforged
  `src/GW/<module>/<module>_bindings.cpp`; EXPOSE the missing FUNCTIONS by wiring the
  native functions that already exist, EXCEPT pure-data accessors (those return structs
  the ctypes contexts already provide). Follow the item async pattern for async strings.

So in the tables below: the "key unbound DATA accessors" columns are mostly NON-issues
(ctypes covers them). Focus on legacy-lib FUNCTION gaps that are operations/decoders.

## Verdict

Under-binding is pervasive and structural: of 24 modules audited, 9 are high-severity, 6 medium, and 9 low/complete. The dominant failure mode is uniform — modules bind their action/getter verbs but drop every `Context*`-returning **data accessor**, and never bind the underlying game-data struct as a `py::class_`. The worst offenders are **quest** (5/15 bound, zero data path, entire legacy PyQuest surface missing), **player** (11/16, all 5 accessors + full world-context snapshot unbound), **agent** (15/25, binds NO agent structs at all), **party** (39/44 but all 5 roster accessors unbound), **skillbar** (live Skillbar/SkillbarSkill state unbound), plus **chat**, **guild**, **trade**, and **ui** (event/callback subscription surface entirely unbound). This forces the Python library back onto ctypes shared-memory scraping for exactly the data these modules already resolve.

## Cross-cutting notes

- **The universal pattern**: every high/medium module fails the same way — free verbs get bound, `Context*`-returning accessors and their structs do not. Fix requires two coordinated steps per module: (1) register the struct as a `py::class_` (def_readonly/def_readwrite, wchar_t arrays converted to str, fixed arrays to lists), and (2) bind the accessor with `py::return_value_policy::reference`. A dict-snapshot alternative works for read-only consumers but breaks write-back and identity; prefer bound structs where the legacy API exposed a class.
- **Mutating accessors** (camera setters, hero-flag writes, trade actions) should be game-thread-enqueued to match legacy semantics, not called inline.
- **Callbacks** (ui Register*/Remove*, item/friend/skillbar callback pairs) need the shared HookEntry-backed trampoline pattern; ui's is the highest-value since it replaces legacy log-polling entirely.
- **Genuinely complete — no action needed**: `agent_recolor`, `dialog`, `merchant`, `packet_sniffer`, `pathing`, `ping` are 1:1 (or superset) ports with no data-accessor gap. `name_obfuscator` and `textures` are effectively complete (unbound items are packet handlers / already-covered helpers); their Wave 3 additions are conveniences, not gaps. `map` and `render` have zero legacy-class gap — only native accessors — so they are lower urgency than their medium tag implies except for `render.get_transform` (load-bearing for the overlay).
